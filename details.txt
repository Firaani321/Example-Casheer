# Detail Arsitektur Aplikasi PrintFlow

Dokumen ini menjelaskan arsitektur teknis dan aliran data dari aplikasi PrintFlow saat ini. Tujuannya adalah untuk memberikan panduan bagi pengembang (atau AI lain) untuk memahami cara kerja sistem sebelum mengimplementasikan fungsionalitas yang lebih kompleks, seperti logika sinkronisasi offline-first.

---

### 1. Tumpukan Teknologi (Tech Stack)

- **Framework**: Next.js 15 (App Router)
- **Bahasa**: TypeScript
- **UI Components**: ShadCN/UI
- **Styling**: Tailwind CSS
- **Database**: Supabase (PostgreSQL)
- **State Management**: React Hooks (`useState`, `useEffect`, `useCallback`). Tidak ada global state manager.
- **Form Handling**: React Hook Form dengan Zod untuk validasi skema.
- **Notifikasi**: `useToast` (custom hook untuk ShadCN Toaster).

---

### 2. Struktur Proyek Kritis

- **`src/app/dashboard/`**: Direktori ini berisi semua halaman utama aplikasi yang dapat diakses pengguna.
  - **`page.tsx`**: Komponen utama untuk setiap rute (misalnya, `projects/page.tsx`).
  - **`layout.tsx`**: Layout utama untuk dasbor yang menyertakan Sidebar dan Header.

- **`src/components/`**: Berisi semua komponen React yang dapat digunakan kembali.
  - **`ui/`**: Komponen dasar dari ShadCN (Button, Card, Dialog, dll.).
  - **`shared/`**: Komponen yang digunakan di banyak halaman (Header, SidebarNav).
  - **`projects/`**, **`expenses/`**, dll.: Komponen spesifik untuk fitur tertentu (misalnya, `project-form.tsx`, `customer-payment-form.tsx`).

- **`src/lib/`**: Berisi file-file inti dan utilitas.
  - **`supabase.ts`**: **Koneksi Klien Supabase.** File ini menginisialisasi dan mengekspor satu-satunya instance klien Supabase yang digunakan di seluruh aplikasi untuk semua interaksi database.
  - **`types.ts`**: **Definisi Tipe TypeScript.** Mendefinisikan semua tipe data utama (Project, Customer, Expense, dll.) yang digunakan di seluruh aplikasi. Sangat penting untuk menjaga konsistensi tipe.
  - **`utils.ts`**: Utilitas umum, terutama fungsi `cn` untuk menggabungkan kelas Tailwind.

- **`db.json`**: File JSON di root proyek yang dimaksudkan sebagai basis data lokal untuk fungsionalitas offline. Saat ini hanya berisi struktur array kosong.

- **`supabase_schema.sql`**: Berisi semua perintah DDL (`CREATE TABLE`, `ALTER TABLE`) untuk mendefinisikan skema database di Supabase. Ini adalah "sumber kebenaran" untuk struktur tabel.

- **`supabase_policies.sql`**: Berisi semua perintah `CREATE POLICY` untuk mengelola Row-Level Security (RLS) di Supabase. Ini mendefinisikan aturan tentang siapa yang dapat mengakses atau memodifikasi data.

---

### 3. Aliran Data & Logika Aplikasi (Arsitektur Saat Ini)

Arsitektur saat ini adalah **online-only** dan sangat bergantung pada koneksi langsung ke Supabase.

1.  **Pengambilan Data (Fetch)**:
    - Setiap halaman (misalnya, `projects/page.tsx`) menggunakan hook `useEffect` dan `useCallback` untuk mengambil data dari Supabase saat komponen dimuat.
    - Fungsi seperti `supabase.from('nama_tabel').select('*')` digunakan untuk mengambil data.
    - Data yang diambil disimpan dalam *state* lokal komponen menggunakan `useState`.
    - Loading UI ditampilkan saat `isLoading` state bernilai `true`.

2.  **Mutasi Data (Insert, Update, Delete)**:
    - Mutasi data dipicu oleh interaksi pengguna (misalnya, menekan tombol "Simpan" pada formulir).
    - Komponen formulir (seperti `project-form.tsx`) memanggil fungsi Supabase secara langsung (misalnya, `supabase.from('projects').insert(...)`).
    - Setelah mutasi berhasil, fungsi *callback* (`onFormSubmit`) yang di-passing sebagai *prop* dari halaman induk akan dipanggil untuk memicu pengambilan data ulang (`fetchData()`), yang kemudian memperbarui UI dengan data terbaru.

3.  **State Management**:
    - Tidak ada *global state*. Setiap halaman mengelola *state*-nya sendiri.
    - Data yang perlu dibagikan antar komponen (seperti `projectData` untuk formulir edit) di-passing melalui *props*.
    - Dialog/modal dikontrol oleh *state* `boolean` (misalnya, `dialogOpen`) di komponen halaman.

4.  **Konvensi Penting**:
    - **Nama Kolom Database**: Semua nama kolom di Supabase (dan di `types.ts`) menggunakan format **`snake_case`** (misalnya, `project_name`, `customer_id`). Ini adalah konvensi yang sangat penting untuk dijaga.
    - **Komponen Klien**: Hampir semua komponen adalah Komponen Klien (`'use client'`) karena mereka berinteraksi langsung dengan hooks React dan Supabase client-side.

---

### 4. Langkah-langkah untuk Implementasi Offline-First (Manual)

Untuk mengimplementasikan arsitektur offline-first, logika di atas perlu diubah secara fundamental.

1.  **Buat Lapisan Akses Data (Data Access Layer)**:
    - Buat satu set fungsi baru (misalnya, dalam `src/lib/data-service.ts`) yang akan menjadi satu-satunya cara aplikasi berinteraksi dengan data. Contoh: `getProjects()`, `addProject(project)`.
    - Di dalam fungsi-fungsi ini, Anda harus mengimplementasikan logika berikut:
        - **Baca Data**: `getProjects()` harus terlebih dahulu mencoba membaca dari `db.json`. Jika `db.json` kosong atau tidak ada, baru ia mengambil dari Supabase dan mengisi `db.json`.
        - **Tulis Data**: `addProject(project)` harus selalu menulis ke `db.json` terlebih dahulu.

2.  **Implementasikan Logika Sinkronisasi**:
    - Buat "antrian" (queue) untuk melacak semua operasi tulis (create, update, delete) yang terjadi saat offline. Antrian ini bisa menjadi bagian dari `db.json`.
    - Buat mekanisme untuk mendeteksi status koneksi internet.
    - Saat aplikasi kembali online, proses antrian tersebut satu per satu, kirimkan setiap operasi ke Supabase.
    - Implementasikan penanganan konflik jika data di Supabase telah berubah. Ini adalah bagian yang paling sulit (misalnya, menggunakan *timestamps* atau *version numbers*).

3.  **Refaktor Komponen**:
    - Ubah semua komponen yang ada agar tidak lagi memanggil Supabase secara langsung. Sebaliknya, mereka harus memanggil fungsi dari *Data Access Layer* yang Anda buat (misalnya, `getProjects()`, `addProject()`).
